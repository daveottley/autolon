# Autolon Next Steps

I think fixed coordinate macros are the wrong next step. They will be brittle the moment the window moves, scales, UI panels change, or DPI/session behavior differs.

The better next step is a task runner with perception-assisted targeting, not raw macro recording.

## Next 2-3 Development Steps

1. Build a Macro/Task Engine

Add a system that can run scripted tasks as state machines:

```text
task:
  find target
  move mouse
  click
  wait
  press key
  verify result
  loop or stop
```

Internally this should not just be "click x=500 y=300." It should support actions like:

```text
click("anvil_button")
wait_until("inventory_open")
press("Escape")
repeat_every(30s)
```

This gives you the foundation for future automation without hardcoding every behavior into Rust.

2. Add a Screen/Window Perception Layer

This is the real next unlock. Autolon needs to know where the game is and where UI elements are.

Start simple:

- Detect the IdleOn window.
- Capture screenshots of that window.
- Locate UI elements by image/template matching.
- Convert found UI element positions into click coordinates.
- Fall back to user calibration when detection fails.

For Wayland, this may require portal-based screenshot/window capture, which can be more annoying than X11, but it is the correct direction.

3. Create User-Calibrated Task Profiles

Before trying full image recognition everywhere, let the user create task profiles:

```text
Task: Collect anvil
Anchor: IdleOn window top-left
Target: anvil collect button
Verification: button changed / popup appeared / timeout
Loop: every 10 minutes
```

The key is using anchors and relative geometry, not absolute screen coordinates. For example:

```text
click at 72% window width, 84% window height
```

or:

```text
find image "claim_button.png", then click center
```

## Best Next Step

Build the task engine plus one concrete task, but keep perception minimal.

I would implement something like:

```text
autolon task run test-click-sequence
```

with a config file such as:

```toml
[name]
task = "sample"

[[steps]]
action = "move"
x_percent = 0.50
y_percent = 0.50

[[steps]]
action = "click"

[[steps]]
action = "key"
key = "Escape"

[[steps]]
action = "wait"
ms = 1000
```

Then evolve it toward:

```toml
[[steps]]
action = "click_image"
image = "claim_button.png"
timeout_ms = 3000
```

That sequence gets Autolon from autoclicker to macro engine without overcommitting to computer vision immediately.

The architecture I would aim for:

```text
autolon-core
  task state machine
  action model
  timing/retry logic

autolon-input
  mouse/keyboard injection

autolon-vision
  screenshot capture
  image matching
  window geometry

autolon-gui
  task list
  run/stop task
  simple calibration UI
```

Avoid full game-playing bot logic for now. The practical next milestone is:

> Autolon can run a named task that clicks window-relative positions, sends keys, loops safely, and can be stopped globally.

After that, add image-based targeting. That will make it resilient enough for resized windows and UI movement.
